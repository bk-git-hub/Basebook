## 문항 1. 과제 수행 과정

아직 개발이 진행 중이라 이 답변은 현재까지의 실제 작업 흐름을 기준으로 정리한 초안입니다. 최종 제출 전에는 구현 마무리, 테스트, 검증 과정을 덧붙여 보완할 예정입니다.

과제를 처음 받았을 때는 바로 화면부터 만들기보다 먼저 `Book Print API가 실제로 어떤 역할을 하는지`와 `Sweetbook이 이미 직접 제공하는 서비스가 무엇인지`를 파악하는 데 집중했습니다. 처음에는 일반적인 B2C 포토북 서비스 아이디어도 떠올렸지만, 공식 사이트와 파트너 사례를 살펴보면서 이 API는 단순 포토북 편집기라기보다 `기존 서비스 안에서 책 제작과 주문을 자동화하는 인프라`에 가깝다고 판단했습니다. 그래서 기획의 출발점을 “예쁜 포토북을 쉽게 만드는 서비스”가 아니라 “기록이 먼저 존재하고, 마지막에 책으로 전환되는 서비스”로 바꾸었습니다.

그 다음에는 아이디어를 여러 번 좁혀 갔습니다. 가족 기록, 크리에이터 아카이브, 커뮤니티 시즌북, 레시피 아카이브 같은 후보를 검토했고, Sweetbook의 기존 서비스와 겹치지 않으면서도 웹앱으로 설득력 있게 보여줄 수 있는 주제를 찾으려고 했습니다. 그 과정에서 최종적으로 선택한 방향이 `베이스북`입니다. 야구팬이 경기 경험을 기록하고, 시즌이 끝나면 그 기록을 한 권의 책으로 주문할 수 있는 서비스로 정의했습니다. 이 과정에서 “핵심은 포토북 제작이 아니라 기록 서비스”라는 기준을 세웠고, LLM 문장 생성처럼 없어도 되는 기능은 MVP에서 제외했습니다.

기획이 어느 정도 정리된 뒤에는 바로 구현에 들어가지 않고, 먼저 설계와 문서화를 해두었습니다. API 분석 문서, 기존 Sweetbook 서비스 분석 문서, 제품 기획서, 프론트엔드 기능 명세, 백엔드 기능 명세, 계약 문서를 따로 만들었습니다. 특히 프론트와 백엔드를 병렬로 개발할 계획이 있었기 때문에, 먼저 `contracts`를 잠가서 요청/응답 형태가 흔들리지 않도록 했습니다. 동시에 `DECISIONS.md`와 프론트/백 마일스톤 로그를 만들어서 어떤 판단을 왜 했는지, 그리고 각 영역이 어디까지 진행됐는지 추적할 수 있게 했습니다.

기술 스택은 처음에는 Next.js 풀스택도 검토했지만, 이번 과제에서 백엔드 역할을 좀 더 명확하게 보여주고 싶어서 `Next.js + Nest.js` 모노레포 구조로 결정했습니다. 프론트엔드는 Next.js, 백엔드는 Nest.js, 데이터 저장은 Prisma + SQLite, 업로드는 Vercel Blob, Book Print API 연동은 Node SDK를 사용하는 방식으로 정리했습니다. 그리고 실제 작업 속도를 높이기 위해 하나의 저장소 안에서 `apps/web`과 `apps/api`를 분리하고, Git worktree를 이용해 프론트엔드와 백엔드를 각각 별도 브랜치와 별도 작업 폴더에서 병렬로 개발할 수 있도록 준비했습니다.

작업 방식은 AI 도구를 적극적으로 활용하되, 무조건 생성된 결과를 받아들이기보다 제가 직접 방향을 계속 조정하는 방식이었습니다. 아이디어를 넓게 탐색할 때는 AI와 함께 후보를 많이 검토했고, 그 다음 단계에서는 공식 API 문서와 실제 서비스 페이지를 기준으로 사실 확인을 했습니다. 또 기획이 정해진 뒤에는 `계획 -> 기술 회의 -> 구현 -> 테스트 -> 리뷰 -> 사용자 검증` 순서를 강제하는 방식으로 운영 규칙도 세웠습니다. 이렇게 한 이유는, AI를 쓰면 속도는 빨라지지만 한 번 잘못된 방향으로 가기 시작하면 오히려 수정 비용이 더 커질 수 있다고 봤기 때문입니다.

시간 배분은 초반에 기획과 분석 비중을 높게 두었습니다. 정확한 최종 비율은 개발이 모두 끝난 뒤 다시 정리해야 하지만, 현재까지는 `API와 기존 서비스 분석 -> 제품 방향 결정 -> 아키텍처/계약 설계 -> 병렬 개발 준비` 순서로 진행했고, 구현 자체는 그 다음 단계에서 프론트와 백엔드를 나눠 동시에 진행하려고 하고 있습니다. 결과적으로 이번 과제는 처음부터 끝까지 “빨리 만드는 것”보다는 “무엇을 왜 만들지 먼저 제대로 결정하고, 그 위에서 병렬로 속도를 내는 것”에 더 가깝게 접근했습니다.
