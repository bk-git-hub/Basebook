## Executive Summary

As of 2026-04-08, Sweetbook is not just a generic photobook printer.

Publicly visible service layers:

1. Consumer D2C custom photo-goods storefront
2. Category-specific themed book templates for travel, baby, wedding, daily life, and more
3. Institution-focused verticals for kindergarten/daycare and school
4. Senior-focused simplified product line
5. Special goods and gift products, including puzzles, calendars, frames, and prints
6. AI-assisted educational storybook platform (`AI스토리교실`)
7. B2B API platform (`Book Print API`)

Planning implication:

- A generic “make a photobook from your photos” service is too close to the existing core business.
- A daycare/kindergarten book service is also too close.
- An AI storybook service for schools is already directly covered by `AI스토리교실`.
- The strongest whitespace is not “another Sweetbook storefront,” but a specialized upstream content-capture or workflow product that uses the print pipeline in a way Sweetbook does not already package for end users.

## Where the Whitespace Still Looks Real

The most promising gaps are not in the final printed object. They are in the source content, workflow trigger, and user job-to-be-done before printing.

### Gap A: Upstream workflow products

Sweetbook already sells the end product well.

A better angle is:

- ingest content from a specific digital workflow
- organize it automatically
- convert it into a print-ready book

Examples of this pattern:

- conversation archive -> book
- community posts -> yearbook
- creator newsletter/archive -> print annual
- family chat/photo stream -> milestone book
- journal/check-in logs -> retrospective book

What makes this different:

- the value is not “we print books”
- the value is “we turn messy digital traces into a coherent printed artifact”

### Gap B: Domain-specific structured records outside existing Sweetbook verticals

Kindergarten and AI story classrooms are already covered, but other repeatable record systems may not be:

- hobby clubs
- sports teams
- fandom communities
- creator communities
- church/small-group memory books
- alumni/reunion records
- team offsite / project retrospective books

This is still risky if framed too broadly, but can work if the source workflow is distinct.

### Gap C: SaaS-like export-to-book tools

Because Sweetbook now publicly promotes `Book Print API by Sweetbook` on the main homepage, one strong angle is:

- design a partner-style app that sits outside the Sweetbook storefront
- convert content from another app/workflow into books

This is meaningfully different from a D2C clone.

## Strategic Takeaway for the Assignment

If we want to look differentiated, the app should not feel like “Sweetbook but smaller.”

It should feel like:

- a specialized product for one specific content source
- with Sweetbook powering the physical book fulfillment behind the scenes

Good framing:

- `X를 책으로 바꾸는 서비스`
- where `X` is not already the default Sweetbook category

Risky framing:

- `사진으로 포토북 만드는 서비스`
- `아이 추억을 포토북으로 만드는 서비스`
- `AI 동화책 서비스`

Those all overlap too directly with what Sweetbook already sells publicly.

## Working Differentiation Rules

A candidate idea is safer if:

1. the source material is not just manual photo upload
2. the user’s main problem is organization or transformation, not printing itself
3. the product has a workflow Sweetbook does not already expose on its public site
4. the value proposition still works even if Sweetbook is replaced with another print backend

A candidate idea is riskier if:

1. it starts from “user uploads photos into a themed editor”
2. it targets kindergarten, school memory books, or AI children’s stories
3. it recreates a gallery, editor, cart, and checkout as the main user value
