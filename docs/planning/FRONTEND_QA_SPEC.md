# Frontend QA Functional Spec

## 목적

이 문서는 `Frontend QA engineer`가 어떤 범위에서 테스트 코드를 작성하고 QA를 수행해야 하는지 정의한다.

핵심 목표는 아래 두 가지다.

- 프론트엔드 핵심 사용자 흐름의 회귀를 막는 테스트를 만든다
- 실제 화면을 기준으로 QA를 수행하고, 발견한 문제를 재현 가능하게 정리하거나 직접 수정한다

---

## 담당 범위

- `apps/web/**` 안의 테스트 코드
- 프론트엔드 QA 중 발견된 버그 수정
- 테스트 편의를 위한 최소한의 selector, aria-label, data attribute 추가
- 로딩, empty, error 상태 검증

직접 담당하지 않는 범위:

- `apps/api/**`
- 루트 설정 파일
- `packages/contracts/**`

계약 관련 변경이 필요하면 CTO 또는 Backend engineer와 먼저 맞춘다.

---

## 우선 테스트 대상

1. 시즌 대시보드 렌더링
2. 경기 기록 작성 흐름
3. 경기 상세 조회 흐름
4. 시즌북 생성 입력 흐름
5. 주문 입력 화면의 기본 상태

---

## 우선 QA 시나리오

### 1. 대시보드

- 기본 더미 데이터가 보이는지
- 최근 기록 목록이 비어 있지 않은지
- 시즌북 만들기 진입 CTA가 있는지

### 2. 경기 기록 작성

- 필수 입력 없이 제출 시 검증이 보이는지
- 경기 선택 후 자동 채움 필드가 반영되는지
- 사진 업로드 UI가 동작하는지
- 저장 성공 후 기대하는 이동 또는 상태 변화가 있는지

### 3. 경기 상세 / 수정

- 저장된 데이터가 올바르게 렌더링되는지
- 수정 화면으로 진입 가능한지
- 수정 저장 후 값이 반영되는지

### 4. 시즌북 생성

- 포함 경기 선택이 가능한지
- 제목/소개 문구 입력이 가능한지
- 견적 요청 버튼 상태가 올바른지

### 5. 주문 화면

- 배송지 입력 필드가 존재하는지
- 필수 입력 검증이 동작하는지
- 완료 상태가 구분되는지

---

## 테스트 전략

1. 컴포넌트/페이지 단위 테스트를 먼저 만든다
2. 해피패스 중심으로 시작한다
3. 그 다음 필수 에러 상태와 empty state를 보강한다
4. QA 과정에서 발견한 버그는 재현 테스트를 먼저 고민한다

---

## 작업 원칙

- 테스트 코드는 작은 커밋 단위로 나눈다
- QA 중 발견한 버그 수정은 가능하면 테스트와 함께 묶는다
- 구현 중인 화면이 크게 바뀌고 있다면 Frontend engineer와 먼저 소통한다
- 사용자 검증 전에는 QA 완료를 최종 완료로 간주하지 않는다

---

## 첫 작업 추천

1. 테스트 러너와 기본 테스트 파일 구조 확인
2. 시즌 대시보드 smoke test 작성
3. 경기 기록 작성 폼의 필수 입력 검증 테스트 작성
4. QA 체크리스트에 따라 수동 점검 시작
