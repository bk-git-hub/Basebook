# 베이스북 발표 브리프

## 서비스 한 줄 소개

`베이스북`은 야구팬이 경기 후 사진과 감상을 기록하고, 시즌이 끝나면 그 기록을 한 권의 책으로 주문할 수 있는 웹 서비스다.

## 핵심 문제

야구팬의 경기 경험은 강한 감정과 추억을 남기지만 대부분 흩어져 사라진다.

- 사진은 사진첩에 남는다
- 감상은 메모앱이나 SNS에 흩어진다
- 시즌 전체의 흐름은 다시 보기 어렵다
- 나중에 책으로 만들고 싶어도 경기별 기억을 다시 모으기 번거롭다

## 핵심 사용자

- 특정 팀을 꾸준히 응원하는 KBO 팬
- 직관을 자주 가는 팬
- 집관도 하지만 시즌 전체를 기록으로 남기고 싶은 팬

## 해결 방식

베이스북은 `포토북 편집기`가 아니라 `경기 경험 기록 서비스`로 시작한다.

사용자는 경기마다 다음을 남긴다.

- 응원팀 / 상대팀
- 날짜 / 구장 / 점수 / 승패
- 한줄 감상
- 오늘의 장면
- 사진
- 자유 메모

이 기록이 시즌 동안 쌓이면, 선택한 기록들을 기반으로 시즌북을 만들고 주문까지 이어진다.

## 왜 Sweetbook API가 필요한가

이 서비스의 가치는 기록만으로도 성립하지만, 사용자가 시즌의 감정과 기억을 `실물 책`으로 보관하는 순간 제품 경험이 완성된다.

Sweetbook API는 이 마지막 단계를 담당한다.

- 책 판형 조회
- 템플릿 조회
- 책 생성
- 사진 업로드
- 표지 생성
- 내지 생성
- 최종화
- 견적 조회
- 주문 생성

## 왜 Sweetbook 직접 서비스와 다른가

Sweetbook 직접 서비스는 결과물을 만드는 경험에 가깝다.

베이스북은 그 전에 있는 과정을 해결한다.

- 시즌 동안 꾸준히 기록하게 만드는 UX
- 경기 메타데이터 자동 보강
- 시즌 단위 누적 구조
- 기록을 책으로 전환하는 자연스러운 흐름

정리하면:

- Sweetbook: 책 제작 fulfillment
- 베이스북: 팬 경험 기록 product

## 차별점 요약

1. 야구 맥락이 있는 기록 UX
2. 경기 메타데이터 자동 보강
3. 시즌 단위 누적 구조
4. 기록에서 책으로 이어지는 자연스러운 전환

## 핵심 사용자 흐름

1. 사용자가 응원팀을 고른다
2. 경기 목록에서 오늘 경기를 선택한다
3. 기본 경기 정보가 자동으로 채워진다
4. 사진과 감상을 입력해 일지를 남긴다
5. 시즌 동안 기록이 누적된다
6. 사용자가 포함할 경기를 골라 시즌북을 만든다
7. Sweetbook API로 견적을 받고 주문한다

## 발표 때 강조할 메시지

### 1. 이 서비스는 포토북 서비스가 아니다

핵심은 기록 습관과 팬 경험의 축적이다.

### 2. Sweetbook API를 가장 자연스럽게 쓰는 지점은 마지막 단계다

책을 만드는 순간만 API를 호출하는 것이 아니라, 기록 데이터가 이미 구조화되어 있기 때문에 책 생성 흐름이 자연스럽다.

### 3. MVP는 과하지 않게 잡았다

- AI 문장 생성 제외
- KBO 데이터는 수동/반자동 보강
- 경기 기록 -> 시즌북 -> 주문의 핵심 해피패스 완성에 집중

## 비즈니스 가능성

첫 수익화 포인트:

- 시즌북 주문
- 플레이오프/특별 에디션 북

중장기 확장:

- 응원팀별 팬 커뮤니티 확장
- 경기 리마인드
- 공동 응원 기록
- 자동 데이터 동기화
- 특별 굿즈성 책 시리즈

## 5분 발표 구성

### 1. 문제 제기

야구팬의 경기 경험은 강한데 기록이 흩어진다.

### 2. 해결책 소개

베이스북은 경기 경험을 기록하고 시즌이 끝나면 책으로 남기는 서비스다.

### 3. 핵심 화면 시연

- 시즌 대시보드
- 경기 기록 작성
- 기록 상세
- 시즌북 생성
- 견적 및 주문

### 4. 기술 구조 설명

- Next.js 웹
- Nest.js API
- Prisma + SQLite
- Vercel Blob
- Sweetbook Book/Order API 연동

### 5. 마무리

기록 서비스가 먼저이고, 책은 그 경험을 완성하는 결과물이라는 점을 다시 강조한다.
