# 베이스북 데모 스크립트

## 데모 목표

짧은 시간 안에 아래 세 가지를 명확하게 보여준다.

1. 사용자가 왜 이 서비스를 쓰는지
2. 왜 Sweetbook API가 필요한지
3. 제품이 기록 서비스로서 먼저 성립한다는 점

## 권장 데모 길이

- 3분 데모
- 2분 기술 설명

## 데모 시나리오

샘플 사용자:

- 응원팀: LG 트윈스
- 시즌 일부 기록: 8~12경기
- 직관/집관 기록 혼합

## 데모 순서

### 1. 랜딩 또는 시즌 홈

화면에서 보여줄 것:

- 서비스 이름 `베이스북`
- 한 줄 설명
- 최근 기록 또는 시즌 요약

말할 내용:

- 이 서비스는 포토북 편집기가 아니라 야구 기록 서비스다
- 사용자는 경기 후 감정과 사진을 남기고, 시즌이 끝나면 책으로 만들 수 있다

### 2. 시즌 대시보드

화면에서 보여줄 것:

- 응원팀
- 누적 경기 수
- 승패 요약
- 최근 일지 리스트

말할 내용:

- 시즌 동안 기록이 계속 쌓이는 구조다
- 한 경기성 앱이 아니라 누적형 제품이다

### 3. 경기 기록 작성 화면

행동:

- 날짜와 응원팀을 선택한다
- 경기 후보 목록이 뜨는 것을 보여준다
- 경기 선택 후 자동 채워진 정보 확인
- 한줄 감상, 오늘의 장면, 사진 입력

말할 내용:

- 여기서 핵심은 경기 메타데이터 자동 보강이다
- 사용자가 점수나 상대팀을 다시 찾지 않아도 기록을 빠르게 남길 수 있다

### 4. 일지 상세 화면

화면에서 보여줄 것:

- 경기 정보
- 사용자가 쓴 메모
- 업로드한 사진

말할 내용:

- 일지는 야구팬의 한 경기 경험을 담는 기본 단위다
- 이 구조화된 데이터가 나중에 책의 페이지가 된다

### 5. 시즌북 생성 화면

행동:

- 포함할 경기 선택
- 책 제목 입력
- 대표 표지 사진 선택
- 시즌 소개 문구 입력

말할 내용:

- 사용자는 모든 경기를 넣을 수도 있고 일부 경기만 골라 책으로 만들 수도 있다
- 이 시점부터 Sweetbook API가 본격적으로 개입한다

### 6. 견적 및 주문 화면

화면에서 보여줄 것:

- 책 요약
- 예상 가격
- 배송지 입력
- 주문 완료 상태

말할 내용:

- 베이스북이 기록을 구조화하고
- Sweetbook API가 실제 책 생성과 주문을 담당한다

## 발표용 핵심 멘트

### 시작 멘트

`베이스북은 야구팬이 경기 경험을 기록하고, 시즌이 끝나면 그 기록을 나만의 시즌북으로 보관할 수 있는 서비스입니다.`

### 문제 정의 멘트

`팬들은 사진은 남기지만, 경기의 감정과 기억은 흩어집니다. 베이스북은 그 흩어진 경험을 경기 단위 기록으로 모읍니다.`

### 차별점 멘트

`이 서비스의 핵심은 포토북 편집이 아니라 기록 UX입니다. 책은 마지막에 붙는 프리미엄 결과물입니다.`

### API 설명 멘트

`기록이 충분히 쌓이면, 그 구조화된 데이터를 바탕으로 Sweetbook API를 사용해 책을 만들고 주문까지 연결합니다.`

### 마무리 멘트

`결국 베이스북은 팬 경험을 디지털에서 끝내지 않고, 시즌의 기억을 실물로 남기게 해주는 서비스입니다.`

## 데모 전 체크

- 샘플 데이터가 로드되는지
- 샘플 이미지가 깨지지 않는지
- 경기 선택 후 자동 채움이 되는지
- 시즌북 생성 해피패스가 끝까지 이어지는지
- 주문 완료 화면이 준비되어 있는지
