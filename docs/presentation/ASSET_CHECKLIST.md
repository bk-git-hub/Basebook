# 발표 자료 준비 체크리스트

## 목적

발표자료와 발표용 보조 자료를 빠르게 만들 수 있도록 필요한 자산을 한 번에 정리한다.

## 슬라이드 권장 구성

1. 문제
2. 서비스 소개
3. 핵심 사용자 흐름
4. 주요 화면
5. Sweetbook API 연동 방식
6. 시스템 아키텍처
7. 비즈니스 확장 가능성

## 캡처해야 할 화면

### 1. 랜딩 또는 시즌 홈

보여줄 포인트:

- 서비스 이름
- 한 줄 설명
- 베이스북의 전체 방향

### 2. 시즌 대시보드

보여줄 포인트:

- 누적 경기 수
- 승패 요약
- 최근 기록
- 시즌북 만들기 CTA

### 3. 경기 기록 작성 화면

보여줄 포인트:

- 경기 선택 UI
- 자동 채워진 경기 정보
- 감상/사진 입력

### 4. 일지 상세 화면

보여줄 포인트:

- 한 경기 기록이 어떻게 구조화되어 있는지
- 사진 + 메모 + 메타데이터

### 5. 시즌북 생성 화면

보여줄 포인트:

- 포함 경기 선택
- 표지 사진 선택
- 책 제목 입력

### 6. 견적 / 주문 화면

보여줄 포인트:

- 예상 가격
- 배송지 입력
- 주문 완료 상태

## 캡처해야 할 기술 자료

### 1. 아키텍처 다이어그램

필요 자료:

- Next.js Web
- Nest.js API
- SQLite
- Vercel Blob
- Sweetbook API

### 2. Sweetbook API 플로우 다이어그램

필요 자료:

- `GET /book-specs`
- `GET /templates`
- `POST /books`
- `POST /books/{bookUid}/photos`
- `POST /books/{bookUid}/cover`
- `POST /books/{bookUid}/contents`
- `POST /books/{bookUid}/finalization`
- `POST /orders/estimate`
- `POST /orders`

### 3. 레포 구조 캡처

필요 자료:

- `apps/web`
- `apps/api`
- `data`
- `docs`

## 발표 전에 준비할 자료

### 필수

- 최종 화면 캡처 6장 이상
- 아키텍처 다이어그램 1장
- Sweetbook API 플로우 다이어그램 1장
- 서비스 소개 한 줄
- 문제 정의 1슬라이드

### 권장

- 30초 이하 짧은 시연 영상
- 주문 해피패스 녹화
- 코드 구조 캡처

## 발표 스크립트용 핵심 문장

### 문제 정의

`야구팬은 경기 사진은 남기지만, 시즌 전체의 감정과 기억은 흩어집니다.`

### 해결책

`베이스북은 경기 경험을 기록하고, 그 기록을 시즌북으로 남길 수 있게 해주는 서비스입니다.`

### 차별점

`핵심은 포토북 편집기가 아니라 경기 경험을 꾸준히 쌓게 만드는 기록 UX입니다.`

### API 설명

`사용자가 쌓은 구조화된 기록을 바탕으로, 마지막 단계에서 Sweetbook API를 통해 실제 책 생성과 주문을 처리합니다.`

## 발표 리허설 체크

- 5분 안에 끝나는지
- 서비스 설명이 너무 길지 않은지
- 기술 설명이 제품 설명을 잡아먹지 않는지
- Sweetbook 직접 서비스와 차이가 분명한지
- `왜 이 서비스가 존재하는가`가 첫 1분 안에 전달되는지

## 자주 나올 질문 예상

### 1. 왜 야구를 선택했는가

답변 포인트:

- 팬 경험이 강하고 반복 기록이 발생한다
- 기록이 쌓일수록 책으로 만들 이유가 분명해진다

### 2. 왜 Sweetbook API가 필요한가

답변 포인트:

- 기록 서비스만으로 끝나지 않고 실물 결과물로 이어져야 제품이 완성된다
- 책 생성과 주문 파이프라인을 안정적으로 연결하기 위해 필요하다

### 3. 왜 AI를 넣지 않았는가

답변 포인트:

- MVP 핵심은 기록 UX와 경기 메타데이터 보강이었다
- 불필요한 비용과 불확실성을 줄이고 핵심 가치에 집중했다

### 4. 경기 데이터는 어떻게 처리하는가

답변 포인트:

- 첫 버전은 더미 데이터와 수동/반자동 보강 중심
- 외부 동기화는 확장 포인트로 남겼다
