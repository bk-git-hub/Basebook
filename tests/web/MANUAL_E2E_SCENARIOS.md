# Frontend Manual E2E Scenarios

이 문서는 Sweetbook 프론트엔드를 처음 써보는 사람도 그대로 따라 하면서 수동으로 기능을 점검할 수 있도록 만든 안내서다.

## 현재 기준 범위

- 핵심 대상:
  - `/season`
  - `/entries/new`
  - `/entries/[id]`
  - `/entries/[id]/edit`
  - `/season-book/new`
  - `/order/[projectId]`
  - `/order/[projectId]/status`
- 참고:
  - `/` 홈 화면은 존재하지만 핵심 QA 동선보다는 소개 성격이 강해 우선순위를 낮춘다.

## 1. 시작 전 준비

### 1-1. 서버 실행

루트 디렉터리에서 아래 둘 중 하나로 서버를 띄운다.

방법 A. 프론트와 API를 같이 실행

```bash
npm run dev
```

방법 B. 터미널을 두 개 열어서 각각 실행

```bash
npm run dev:api
```

```bash
npm run dev:web
```

기본 주소:

- 프론트: `http://localhost:3000`
- API: `http://localhost:4000`

### 1-2. 먼저 확인할 것

브라우저에서 먼저 `http://localhost:3000/season` 을 연다.

- 시즌 대시보드가 보이면 정상 시작이다.
- 에러 카드가 보이면 API가 안 떠 있거나 `GET /entries` 에 문제가 있는 상태다.
- 빈 상태 카드가 보이면 연결은 됐지만 기록 데이터가 아직 없는 상태다.

### 1-3. 이 문서를 읽는 법

각 시나리오는 아래 순서로 구성된다.

1. 어디로 이동하는지
2. 무엇을 클릭하거나 입력하는지
3. 화면에서 어떤 변화가 보여야 하는지
4. 이상이 있다면 무엇을 버그로 보면 되는지

## 2. 시나리오 A. 시즌 대시보드 확인

### 목표

`GET /entries` 로 받은 기록 목록이 최신 시즌 기준 요약 카드와 기록 리스트에 반영되는지 본다.

### 따라 하기

1. `http://localhost:3000/season` 으로 이동한다.
2. 상단 카드에 팀 로고, 팀 이름, 시즌 연도가 보이는지 확인한다.
3. 우측 `직관 승률` 카드에 승률과 `승/무/패` 요약이 보이는지 본다.
4. 아래 요약 카드의 `시즌 기록`, `승리`, `패배`, `무승부` 값을 본다.
5. `전체 기록` 섹션에서 카드가 한 개 이상 보이는지 확인한다.
6. 기록 카드 하나를 클릭한다.

### 기대 결과

1. 최신 시즌 기준 요약 숫자와 기록 목록이 함께 보인다.
2. 기록 카드에는 결과 배지, 날짜, 팀 이름, 한 줄 감상, 점수가 함께 보인다.
3. 기록 카드를 누르면 해당 상세 페이지 `/entries/{id}` 로 이동한다.

### 이상 징후

- 대시보드가 흰 화면으로 멈춘다.
- 요약 숫자는 있는데 최근 기록이 전혀 보이지 않는다.
- 카드 클릭 시 상세 화면으로 이동하지 않는다.

## 3. 시나리오 B. 대시보드 empty / error 상태 확인

### 목표

첫 사용자나 서버 장애 상황에서 화면이 이해 가능한 상태를 유지하는지 본다.

### 따라 하기: empty state

1. 데이터가 비어 있는 환경에서 `http://localhost:3000/season` 을 연다.

### empty state 기대 결과

1. `아직 시즌 기록이 없습니다` 제목이 보인다.
2. `새 일지 남기기` CTA가 보인다.

### 따라 하기: error state

1. API 서버를 끈다.
2. `http://localhost:3000/season` 을 새로고침한다.

### error state 기대 결과

1. `시즌 기록을 불러오지 못했습니다` 제목이 보인다.
2. 백엔드 연결 실패 같은 에러 메시지가 카드 안에 남아 있다.
3. 브라우저 기본 에러 페이지로 튕기지 않는다.

## 4. 시나리오 C. 기록 상세 화면 읽기

### 목표

`GET /entries/:id` 응답이 상세 페이지에 자연스럽게 표시되는지 확인한다.

### 따라 하기

1. `/season` 에서 최근 기록 카드 하나를 클릭한다.
2. 상단 제목과 날짜를 확인한다.
3. 우측 상단의 `이 기록 수정하기` 버튼을 확인한다.
4. `경기 요약`, `기록 메타`, `한 줄 감상`, `사진` 섹션을 읽어본다.
5. `시즌 대시보드로 돌아가기` 버튼을 누른다.

### 기대 결과

1. 경기 제목과 결과 배지가 보인다.
2. `이 기록 수정하기` 버튼이 `/entries/{id}/edit` 로 이어진다.
3. 사진이 있으면 `원본 열기` 링크가 보인다.
4. 사진이 없으면 사진이 없다는 안내 문구가 자연스럽게 보인다.
5. 돌아가기 버튼을 누르면 `/season` 으로 이동한다.

## 5. 시나리오 D. 존재하지 않는 기록 상세 화면 확인

### 목표

없는 id일 때 404 화면이 명확한지 확인한다.

### 따라 하기

1. 주소창에 `http://localhost:3000/entries/not-a-real-entry-id` 를 입력한다.

### 기대 결과

1. `요청한 기록을 찾지 못했습니다` 제목이 보인다.
2. 404 안내 문구가 보인다.
3. `시즌 대시보드로 이동` 버튼을 누르면 `/season` 으로 돌아간다.

## 6. 시나리오 E. 새 기록 작성 화면 기본 검증

### 목표

사용자가 `/entries/new` 에서 필요한 입력을 이해하고, 잘못된 입력에 대해 즉시 안내받는지 본다.

### 따라 하기

1. `http://localhost:3000/entries/new` 로 이동한다.
2. 상단에 `오늘의 직관 기록을 남겨보세요` 제목이 보이는지 확인한다.
3. 아무 값도 크게 수정하지 않고 `새 기록 저장` 버튼을 누른다.

### 기대 결과

1. `홈으로 돌아가기` 링크가 보인다.
2. `경기 기본 정보`, `결과와 관람 정보`, `감상 작성`, `사진 업로드` 구역이 보인다.
3. `결과` 기본값이 `미정` 으로 보인다.
4. `한 줄 감상은 비워둘 수 없습니다.` 문구가 보인다.
5. 직관 상태라면 `직관 기록에는 경기장을 선택해 주세요.` 문구도 함께 보인다.
4. 페이지 이동 없이 현재 화면에 남는다.

### 추가 검증

1. `응원 팀`과 `상대 팀`을 같게 맞춘 뒤 저장한다.
2. 점수를 한쪽만 입력한 뒤 저장한다.
3. 모바일 폭으로 줄였을 때 팀 선택이 dropdown 중심으로 보이는지 확인한다.
4. `관람 형태`를 `TV 시청` 또는 `모바일 시청` 으로 바꿨을 때 `경기장`, `좌석` 입력이 사라지는지 확인한다.

### 기대 결과

1. `응원 팀과 상대 팀은 같을 수 없습니다.` 문구가 보인다.
2. `점수는 두 팀 모두 입력하거나 둘 다 비워 주세요.` 문구가 보인다.
3. 모바일에서는 선택된 팀 배지와 dropdown이 함께 보인다.
4. 원격 시청 상태에서는 제목과 설명이 경기 기록 기준으로 바뀌고, 직관 전용 입력이 숨겨진다.

## 7. 시나리오 F. 사진 업로드와 저장 버튼 상태 확인

### 목표

사진 업로드 중 저장 버튼이 막히고, 업로드 후 개별 제외가 가능한지 본다.

### 따라 하기

1. `/entries/new` 에서 `사진 선택` 영역을 누른다.
2. 이미지 파일 한 장 이상을 고른다.
3. 업로드 중 문구와 저장 버튼 상태를 본다.
4. 업로드가 끝나면 업로드된 파일 카드에서 `이 사진 제외` 버튼을 눌러 본다.

### 기대 결과

1. 업로드 중 `업로드 중:` 문구가 보이고 저장 버튼은 비활성화된다.
2. 업로드 완료 후 업로드된 파일 이름과 `업로드 결과 열기` 링크가 보인다.
3. `이 사진 제외` 를 누르면 해당 파일이 목록에서 빠지고 안내 문구가 갱신된다.

## 8. 시나리오 G. 새 기록 생성 성공 흐름

### 목표

사용자가 새 기록을 저장하면 `POST /entries` 호출 뒤 상세 화면으로 이동하는지 확인한다.

### 따라 하기

1. `/entries/new` 에서 필수 항목을 채운다.
2. 필요하면 `상세 메모`, 점수, 경기장, 오늘의 선수도 채운다.
3. 필요하면 사진을 업로드한다.
4. `새 기록 저장` 버튼을 누른다.

### 기대 결과

1. 저장 중 버튼 문구가 `저장 중...` 으로 바뀐다.
2. `기록을 저장했습니다. 생성된 상세 화면으로 이동합니다.` 안내가 보일 수 있다.
3. 곧바로 `/entries/{newId}` 상세 화면으로 이동한다.
4. 상세 화면에 방금 입력한 값이 반영돼 있다.

## 9. 시나리오 H. 수정 화면 진입과 변경 없는 저장

### 목표

수정 화면이 기존 데이터를 채워주고, 변경이 없을 때 PATCH를 보내지 않는지 본다.

### 따라 하기

1. 상세 페이지에서 `이 기록 수정하기` 버튼을 누른다.
2. 아무 필드도 바꾸지 않는다.
3. `수정 내용 저장` 버튼을 누른다.

### 기대 결과

1. 기존 값이 입력칸에 미리 채워져 있다.
2. 페이지 이동은 일어나지 않는다.
3. 저장 요청 때문에 상세 페이지로 튕기지 않는다.

## 10. 시나리오 I. 기록 수정 성공 흐름

### 목표

바뀐 필드만 `PATCH /entries/:id` 로 보내고, 저장 후 상세 화면으로 돌아가는지 확인한다.

### 따라 하기

1. `/entries/{id}/edit` 로 이동한다.
2. `한 줄 감상` 또는 `상세 메모`, `오늘의 선수`, `경기장` 중 하나 이상을 바꾼다.
3. `수정 내용 저장` 버튼을 누른다.

### 기대 결과

1. 저장 중 버튼 문구가 `저장 중...` 으로 바뀐다.
2. `수정 내용을 저장했습니다. 상세 화면으로 이동합니다.` 안내가 보일 수 있다.
3. 저장 후 상세 화면으로 돌아간다.
4. 상세 화면에서 바꾼 값이 반영돼 있다.

## 11. 시나리오 J. 시즌북 기록 선택과 견적 생성

### 목표

`/season-book/new` 에서 기록 선택, 커버 설정, 견적 생성이 자연스럽게 이어지는지 본다.

### 따라 하기

1. `http://localhost:3000/season-book/new` 로 이동한다.
2. 상단 제목 `시즌북에 담을 경기 기록을 고르세요` 를 확인한다.
3. `전체 선택` 버튼을 눌러 기록을 모두 고른다.
4. 오른쪽 `선택 요약` 에 기록 수와 첨부 사진 수가 반영되는지 본다.
5. `시즌북 제목` 과 필요하면 `소개문` 을 입력한다.
6. `커버 사진 URL` 을 직접 입력하거나 `선택 기록의 첫 사진을 커버로 사용` 버튼을 누른다.
7. `시즌북 견적 생성` 버튼을 누른다.

### 기대 결과

1. 선택한 기록 수가 오른쪽 요약에 바로 반영된다.
2. 선택 해제 후 다시 선택해도 요약이 맞게 바뀐다.
3. 견적 생성 중 버튼 문구가 `견적 생성 중...` 으로 바뀔 수 있다.
4. 성공 시 `견적 생성이 완료되었습니다.` 안내와 페이지 수, 예상 금액, 크레딧 정보가 보인다.
5. `주문 화면으로 이동` 링크가 생성된다.

### 실패 관찰 포인트

1. 선택 기록이 없으면 선택 관련 오류가 보여야 한다.
2. 제목이 비어 있으면 제목 오류가 보여야 한다.
3. 커버 사진 URL이 없으면 커버 관련 오류가 보여야 한다.

## 12. 시나리오 K. 시즌북 주문 접수

### 목표

견적 완료 뒤 `/order/[projectId]` 에서 배송 정보를 입력하고 주문 완료 화면까지 이어지는지 확인한다.

### 따라 하기

1. `/season-book/new` 에서 견적을 만든 뒤 `주문 화면으로 이동` 을 누른다.
2. 상단 제목 `시즌북을 받을 배송 정보를 입력하세요` 를 확인한다.
3. 오른쪽 요약의 `견적 확인 완료`, `주문 전 확인`, 페이지 수, 예상 금액을 읽는다.
4. 아래 값을 입력한다.
   - `수령인 이름`
   - `전화번호`
   - `우편번호`
   - `기본 주소`
   - 필요하면 `상세 주소`
5. `시즌북 주문 접수` 버튼을 누른다.

### 기대 결과

1. 필수 항목이 비어 있으면 각 필드 아래에 오류 문구가 보인다.
2. 제출 중 버튼 문구가 `주문 접수 중...` 으로 바뀔 수 있다.
3. 성공 시 `주문이 접수되었습니다.` 안내가 보인다.
4. 완료 카드에 `CONFIRMED`, `ORDERED` 같은 상태 값이 보인다.
5. `주문 진행 상태 보기` 링크가 나타난다.

## 13. 시나리오 L. 주문 진행 상태 조회

### 목표

`/order/[projectId]/status` 화면이 최신 주문 상태를 읽기 화면으로 보여주는지 확인한다.

### 따라 하기

1. 주문 완료 후 `주문 진행 상태 보기` 링크를 누른다.
2. 또는 `/order/{projectId}/status` 로 직접 이동한다.
3. 상단 제목 `시즌북 제작이 어디까지 진행됐는지 확인하세요` 를 확인한다.
4. `현재 요약` 카드에서 프로젝트 상태, 주문 상태, 상태 기준, 최종 갱신 시각을 본다.
5. `상태 새로고침` 버튼을 눌러 재진입이 가능한지 본다.

### 기대 결과

1. 주문 진행 상태 페이지가 열리고, 현재 상태 요약이 보인다.
2. 새로고침 버튼은 같은 status 화면으로 재진입한다.
3. 에러 시에는 `주문 진행 상태를 불러오지 못했습니다` 카드가 보이고 `다시 시도하기` 버튼이 제공된다.

### 주의

- 이 화면은 현재 구현이 빠르게 변하는 영역이라 자동화보다 수동 재확인을 우선한다.

## 14. 빠른 추천 동선

시간이 없을 때는 아래 순서만 봐도 핵심 흐름을 거의 확인할 수 있다.

1. `/season`
2. 최근 기록 클릭 후 `/entries/{id}`
3. `이 기록 수정하기` 클릭 후 수정 저장
4. `/entries/new` 에서 새 기록 생성
5. `/season-book/new` 에서 전체 선택 후 견적 생성
6. `/order/{projectId}` 에서 배송지 입력 후 주문 완료
7. `/order/{projectId}/status` 진입

## 15. 현재 수동 확인이 더 필요한 영역

- 주문 진행 상태의 세부 타임라인과 예외 케이스
- 주문 취소와 배송지 변경 흐름
- 시즌북 견적/주문 실패 케이스 전체
- 백엔드 demo data 품질과 문자 인코딩 상태
