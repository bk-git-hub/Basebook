# Frontend Manual E2E Scenarios

이 문서는 Sweetbook 프론트엔드를 처음 써보는 사람도 그대로 따라 하면서 수동으로 기능을 점검할 수 있도록 만든 안내서다.

기준:

- 현재 구현된 프론트 화면만 다룬다.
- 테스트 대상은 `/season`, `/entries/new`, `/entries/[id]`, `/entries/[id]/edit` 이다.
- `/season-book/new`, `/order/[projectId]` 는 아직 구현 전이라 이 문서에 포함하지 않는다.
- `/` 홈 화면은 아직 Next.js 기본 화면이라 QA 대상에서 제외한다.

---

## 1. 시작 전 준비

### 1-1. 서버 실행

루트 디렉터리에서 아래 둘 중 하나로 서버를 띄운다.

방법 A. 프론트와 API를 같이 실행

```bash
npm run dev
```

방법 B. 터미널을 두 개 열어서 각각 실행

```bash
npm run dev:api
```

```bash
npm run dev:web
```

기본 주소:

- 프론트: `http://localhost:3000`
- API: `http://localhost:4000`

### 1-2. 확인 포인트

브라우저에서 먼저 `http://localhost:3000/season` 을 연다.

- 시즌 대시보드가 보이면 정상 시작이다.
- 에러 카드가 보이면 API가 안 떠 있거나 `GET /entries` 에 문제가 있는 상태다.
- 빈 상태 카드가 보이면 연결은 됐지만 기록 데이터가 아직 없는 상태다.

### 1-3. 이 문서를 읽는 법

각 시나리오는 아래 순서로 구성되어 있다.

1. 어디로 이동하는지
2. 무엇을 클릭하거나 입력하는지
3. 화면에서 어떤 변화가 보여야 하는지
4. 이상이 있다면 어떤 점을 버그로 보면 되는지

## 2. 시나리오 A. 시즌 대시보드가 정상적으로 보이는지 확인

### 목표

`GET /entries` 로 받은 기록 목록이 시즌 요약 카드와 최근 기록 리스트에 반영되는지 본다.

### 따라 하기

1. 브라우저 주소창에 `http://localhost:3000/season` 을 입력한다.
2. 상단 카드에 `Season Dashboard` 배지와 시즌/응원 팀 제목이 보이는지 본다.
3. 아래 요약 카드 4개에서 `총 경기 수`, `승리`, `패배`, `무승부` 숫자를 확인한다.
4. `최근 기록` 섹션에서 기록 카드가 한 개 이상 보이는지 확인한다.
5. 최근 기록 카드 하나를 클릭한다.

### 기대 결과

1. 페이지가 흰 배경 카드형 레이아웃으로 렌더링된다.
2. 기록이 있으면 요약 숫자와 최근 기록 목록이 보인다.
3. 최근 기록 카드에는 아래 정보가 함께 보인다.
   - 결과 배지 `승`, `패`, `무`, `미정`
   - 경기 날짜
   - `LG 트윈스 vs 두산 베어스` 같은 팀 이름
   - 한 줄 감상
   - 우측의 `Final Score`
4. 기록 카드를 누르면 해당 기록의 상세 페이지 `/entries/{id}` 로 이동한다.

### 이상 징후

- `/season` 에 접속했는데 브라우저가 멈추거나 흰 화면만 보인다.
- 요약 숫자가 보이는데 최근 기록 카드가 전혀 보이지 않는다.
- 기록 카드를 눌러도 상세 화면으로 이동하지 않는다.

## 3. 시나리오 B. 시즌 대시보드의 빈 상태와 에러 상태 확인

### 목표

첫 사용자나 서버 장애 상황에서 화면이 최소한 이해 가능한 상태를 보여주는지 본다.

### 따라 하기: 빈 상태

1. 백엔드 데이터가 비어 있는 환경에서 `http://localhost:3000/season` 을 연다.

### 빈 상태 기대 결과

1. `표시할 시즌 기록이 아직 없습니다` 라는 제목이 보인다.
2. `GET /entries 호출은 성공했지만...` 으로 시작하는 안내 문구가 보인다.
3. 대시보드 카드 대신 빈 상태 카드만 보여야 한다.

### 따라 하기: 에러 상태

1. API 서버를 끈다.
2. 브라우저에서 `http://localhost:3000/season` 을 새로고침한다.

### 에러 상태 기대 결과

1. `시즌 기록을 불러오지 못했습니다` 라는 제목이 보인다.
2. 에러 메시지와 함께 현재 API 주소가 `http://localhost:4000` 임을 알려주는 안내가 보인다.
3. 앱이 죽지 않고 카드형 에러 화면을 유지해야 한다.

### 이상 징후

- 빈 상태인데 에러처럼 보이거나, 에러인데 빈 상태처럼 보인다.
- API를 껐을 때 브라우저 기본 에러 페이지로 튕긴다.

## 4. 시나리오 C. 기록 상세 화면 읽기

### 목표

`GET /entries/:id` 응답이 상세 페이지에 자연스럽게 표시되는지 확인한다.

### 따라 하기

1. `/season` 의 최근 기록 카드 하나를 클릭해 상세 페이지로 들어간다.
2. 상단에 경기 제목과 날짜가 보이는지 확인한다.
3. 우측 상단의 `이 기록 수정하기` 버튼을 확인한다.
4. 아래 `경기 요약`, `기록 메타`, `한 줄 감상`, `사진` 섹션을 순서대로 읽어본다.
5. 화면 하단의 `시즌 대시보드로 돌아가기` 버튼을 누른다.

### 기대 결과

1. 상단에 `Entry Detail` 배지와 함께 `LG 트윈스 vs ...` 형식의 제목이 보인다.
2. 결과 배지와 `이 기록 수정하기` 버튼이 함께 보인다.
3. `경기 요약` 에서 점수, 관람 형태, 오늘의 선수, 경기장, 좌석을 볼 수 있다.
4. `기록 메타` 에서 `Entry ID`, 생성 시각, 마지막 수정 시각을 볼 수 있다.
5. `한 줄 감상` 과 `상세 메모` 에 저장된 문장이 보여야 한다.
6. 사진이 없으면 `첨부된 사진이 없습니다...` 문구가 보인다.
7. `시즌 대시보드로 돌아가기` 를 누르면 다시 `/season` 으로 돌아간다.

### 이상 징후

- 상세 화면 진입 후 제목이나 감상 텍스트가 비어 있다.
- `이 기록 수정하기` 버튼이 보이지 않는다.
- 하단 돌아가기 버튼이 눌리지 않거나 잘못된 주소로 이동한다.

## 5. 시나리오 D. 존재하지 않는 기록 상세 화면 확인

### 목표

없는 기록 id일 때 404 화면이 명확하게 보이는지 확인한다.

### 따라 하기

1. 브라우저 주소창에 존재하지 않을 것 같은 id를 직접 입력한다.
2. 예시: `http://localhost:3000/entries/not-a-real-entry-id`

### 기대 결과

1. `요청한 기록을 찾지 못했습니다` 라는 제목이 보인다.
2. 본문에 ``GET /entries/:id`가 404를 반환한 상태입니다.` 라는 안내가 보인다.
3. `시즌 대시보드로 이동` 버튼이 보인다.
4. 그 버튼을 누르면 `/season` 으로 이동한다.

### 이상 징후

- 없는 id인데도 빈 상세 화면이 뜬다.
- 404 대신 일반 서버 에러처럼 보인다.

---

## 6. 시나리오 E. 새 기록 작성 화면 진입과 기본 검증

### 목표

처음 들어온 사용자가 `/entries/new` 에서 어떤 입력이 필요한지 이해하고, 잘못된 입력에 대해 즉시 안내를 받는지 본다.

### 따라 하기

1. 브라우저 주소창에 `http://localhost:3000/entries/new` 를 입력한다.
2. 상단에 `POST /entries 생성 흐름` 제목이 보이는지 확인한다.
3. 아무 값도 크게 수정하지 말고 `한 줄 감상` 을 비운 상태로 둔다.
4. 하단의 `새 기록 저장` 버튼을 누른다.

### 기대 결과

1. 화면 상단 우측에 `시즌 대시보드로 돌아가기` 버튼이 보인다.
2. 폼은 `경기 기본 정보`, `결과와 관람 정보`, `경기 후보 조회`, `감상 작성`, `생성 요청 미리보기` 로 나뉘어 보인다.
3. `한 줄 감상은 비워둘 수 없습니다.` 라는 검증 문구가 보여야 한다.
4. 페이지 이동 없이 현재 화면에 그대로 남아야 한다.

### 추가 검증 1: 같은 팀 선택

1. `응원 팀` 과 `상대 팀` 을 같은 팀으로 맞춘다.
2. `새 기록 저장` 을 누른다.

기대 결과:

1. `응원 팀과 상대 팀은 같을 수 없습니다.` 문구가 보인다.

### 추가 검증 2: 점수 한쪽만 입력

1. `우리 팀 점수` 에만 숫자를 넣고 `상대 팀 점수` 는 비운다.
2. `새 기록 저장` 을 누른다.

기대 결과:

1. 두 점수 입력칸 아래에 모두 `점수는 두 팀 모두 입력하거나 둘 다 비워 주세요.` 문구가 보인다.

### 이상 징후

- 필수 검증이 동작하지 않고 바로 저장된다.
- 오류 문구는 떴는데 저장 요청도 같이 나가서 상세 화면으로 이동한다.

## 7. 시나리오 F. 경기 후보 조회와 자동 채움 확인

### 목표

`GET /games` 를 이용한 후보 조회가 동작하고, 사용자가 후보를 선택했을 때 생성 폼 일부가 자동으로 채워지는지 확인한다.

### 따라 하기

1. `/entries/new` 화면에서 `관람 날짜` 와 `응원 팀` 을 적절히 고른다.
2. `경기 후보 불러오기` 버튼을 누른다.
3. 후보가 뜨면 카드 하나를 클릭한다.

### 기대 결과

1. 버튼을 누르는 동안 버튼 문구가 `경기 불러오는 중...` 으로 바뀔 수 있다.
2. 후보가 있으면 `후보를 확인한 뒤 원하는 경기를 선택하면 폼이 자동으로 채워집니다.` 안내가 보인다.
3. 후보 카드에는 상태 배지, 날짜, 소스, 팀 이름, 구장, 결과, `gameId`, 점수 정보가 보인다.
4. 후보 카드 하나를 누르면 선택된 카드가 진하게 강조된다.
5. 선택 직후 `선택한 경기 후보를 폼에 반영했습니다. 필요한 항목은 계속 수정할 수 있습니다.` 안내가 보여야 한다.
6. 아래 항목이 자동으로 바뀌는지 확인한다.
   - `gameId`
   - `시즌 연도`
   - `관람 날짜`
   - `응원 팀`
   - `상대 팀`
   - 점수
   - 결과
   - 경기장

### 후보가 없을 때 기대 결과

1. `조회 조건에 맞는 경기 후보가 없습니다. 날짜나 응원 팀을 바꿔 다시 시도해 주세요.` 안내가 보인다.
2. 앱이 깨지지 않고 폼을 계속 수정할 수 있어야 한다.

### 이상 징후

- 후보 조회 후 카드가 보이는데 클릭해도 폼 값이 바뀌지 않는다.
- 후보가 없을 때 화면이 비정상적으로 비거나 버튼이 사라진다.

## 8. 시나리오 G. 새 기록 생성 성공 흐름

### 목표

사용자가 새 기록을 저장하면 `POST /entries` 호출 뒤 상세 화면으로 이동하는지 확인한다.

### 따라 하기

1. `/entries/new` 화면으로 이동한다.
2. 아래 값들을 적절히 채운다.
   - `한 줄 감상`
   - 필요하면 `상세 메모`
   - 필요하면 `우리 팀 점수`, `상대 팀 점수`
3. 우측 `생성 요청 미리보기` 카드에서 JSON이 현재 입력과 맞는지 한 번 읽어본다.
4. `새 기록 저장` 버튼을 누른다.

### 기대 결과

1. 저장 중에는 버튼 문구가 `저장 중...` 으로 바뀐다.
2. 잠시 뒤 초록색 성공 메시지 `기록을 저장했습니다. 생성된 상세 화면으로 이동합니다.` 가 보일 수 있다.
3. 곧바로 새 상세 페이지 `/entries/{newId}` 로 이동해야 한다.
4. 상세 페이지에 방금 입력한 `한 줄 감상`, `상세 메모`, 점수 등이 반영돼 있어야 한다.

### 이상 징후

- 성공 메시지가 떴는데 페이지가 이동하지 않는다.
- 상세 페이지로 갔지만 방금 입력한 값이 반영되지 않는다.
- 미리보기 JSON과 실제 저장 결과가 크게 다르다.

---

## 9. 시나리오 H. 수정 화면 진입과 변경 없는 저장

### 목표

수정 화면이 기존 데이터를 잘 불러오고, 아무것도 바꾸지 않았을 때는 PATCH를 보내지 않는지 확인한다.

### 따라 하기

1. 상세 페이지에서 `이 기록 수정하기` 버튼을 누른다.
2. 수정 화면 상단에 `Entry Edit` 배지와 현재 한 줄 감상이 제목으로 보이는지 확인한다.
3. 아무 필드도 수정하지 않는다.
4. `수정 내용 저장` 버튼을 누른다.

### 기대 결과

1. 수정 화면 진입 시 기존 값이 입력칸에 미리 채워져 있어야 한다.
2. 저장 후 페이지 이동은 일어나지 않는다.
3. 초록색 안내 문구 `변경된 항목이 없어 저장 요청을 보내지 않았습니다.` 가 보여야 한다.
4. 우측 `PATCH 미리보기` 는 빈 객체 `{}` 여야 한다.

### 이상 징후

- 기존 값이 비어 있거나 일부만 채워져 있다.
- 아무것도 안 바꿨는데도 상세 화면으로 이동한다.

## 10. 시나리오 I. 기록 수정 성공 흐름

### 목표

변경한 필드만 `PATCH /entries/:id` 로 보내고, 저장 후 상세 화면으로 돌아가는지 확인한다.

### 따라 하기

1. 상세 페이지에서 `이 기록 수정하기` 버튼을 눌러 수정 화면으로 들어간다.
2. 아래 중 하나 이상을 바꾼다.
   - `한 줄 감상`
   - `상세 메모`
   - `오늘의 선수`
   - `경기장`
   - `좌석`
   - 점수
3. 우측 `PATCH 미리보기` 카드에서 바꾼 필드만 JSON에 들어가는지 확인한다.
4. `수정 내용 저장` 버튼을 누른다.

### 기대 결과

1. 저장 중에는 버튼 문구가 `저장 중...` 으로 바뀐다.
2. 초록색 안내 문구 `수정 내용을 저장했습니다. 상세 화면으로 이동합니다.` 가 보일 수 있다.
3. 저장 후 자동으로 상세 페이지로 돌아간다.
4. 상세 페이지에서 바꾼 값이 반영돼 있어야 한다.
5. `생성 시각` 은 그대로이고, `마지막 수정` 시각은 달라질 수 있다.

### 이상 징후

- `PATCH 미리보기` 에 안 바꾼 필드까지 대량으로 들어간다.
- 저장은 됐는데 상세 화면 값이 예전 그대로다.
- 수정 후 잘못된 기록 상세 페이지로 이동한다.

## 11. 시나리오 J. 수정 화면 실패 케이스

### 목표

수정 화면에서도 잘못된 입력을 친절하게 막는지 확인한다.

### 따라 하기

1. `/entries/{id}/edit` 로 들어간다.
2. `한 줄 감상` 을 비운다.
3. `수정 내용 저장` 을 누른다.

### 기대 결과

1. `한 줄 감상은 비워둘 수 없습니다.` 문구가 보인다.
2. 저장 요청 없이 현재 화면에 남아 있어야 한다.

### 추가 검증

1. `응원 팀` 과 `상대 팀` 을 같게 맞춘 뒤 저장한다.
2. 점수를 한쪽만 입력한 뒤 저장한다.

기대 결과:

1. 생성 화면과 동일하게 팀 중복 오류와 점수 쌍 오류가 보여야 한다.

### 이상 징후

- 수정 화면에서는 생성 화면과 달리 검증이 빠져 있다.
- 오류 문구가 보였는데도 상세 화면으로 이동한다.

## 12. 빠른 추천 동선

시간이 없을 때는 아래 순서로만 봐도 핵심 흐름을 거의 확인할 수 있다.

1. `/season` 에서 최근 기록 카드가 보이는지 확인
2. 최근 기록 카드 클릭 후 상세 페이지 진입 확인
3. `이 기록 수정하기` 클릭 후 수정 화면 진입 확인
4. 수정 화면에서 `한 줄 감상` 하나 바꾸고 저장
5. 주소창에 `/entries/new` 입력
6. 새 기록 하나 작성하고 `새 기록 저장`
7. 생성된 상세 페이지에서 입력한 값 확인
